Two threads, one package

The copilot is its own package mirroring exporters/. App owns a handle + drains per frame. GL lives on the main thread; the worker never touches moderngl/imgui — it reaches GL only through the bridge.

• capabilities imports nothing → App implements it → edge is app → copilot, never reverse (the cycle-break).

• llm/api imports no SDK → the openai SDK stays inside openrouter.py; the seam is provider-neutral & fake-able for tests.

• tools imports capabilities, never App — else app → copilot → app would cycle.

The five seams

SeamModuleBoundary it drawsShape
Acapabilities.py App ↔ tool layer frozen dataclass of bound callables (GL-free + GL-touching halves). Built by App; imported by tools. The cycle-break.
Bllm/api.py agent ↔ LLM provider sync LLMClient Protocol (stream / complete_text / complete_structured) + neutral dataclasses. SDK hidden in the impl.
Cbridge.py worker → main (GL) run_on_main(fn) blocks the worker; drain() on main runs it; cancel_all() on teardown. The one new mechanism.
D2× queue.Queue (in session.py) worker ↔ UI (async) request queue (UI→worker, blocking drain) + event queue (worker→UI, non-blocking pump per frame). Exporter pattern.
Estate.py UI ↔ everything else plain-dataclass transcript + render-state. UI reads it (dumb); pump_events mutates it. Keeps imgui off the worker.

The two cross-thread flows (what the seams carry)

GL op (Seam C): worker tool needs GL → bridge.run_on_main(fn) (worker blocks) → main-thread drain() runs fn with live GL, sets event → worker unblocks with the result. Teardown: cancel_all() before join() releases any blocked worker.
Turn + streaming + action-required (Seams D/E): UI enqueues a user turn (request queue) → worker runs agent loop, pushes events (token deltas, tool-status, pending-action) to the event queue → main-thread pump_events() folds them into state → the tab draws state. On a pending-action: the loop blocks on the request queue until the UI enqueues the user's answer — same two-queue seam, no new mechanism.
